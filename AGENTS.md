# RimWorld AI Translator 프로젝트 지침

## 시작 절차

1. `docs/PROJECT_STATE.md`, `docs/ROADMAP.md`, `docs/QUALITY_GATES.md`, `docs/ARCHITECTURE.md`를 읽는다.
2. `git status --short --branch`, staged/unstaged diff와 새 파일을 확인한다. 문서의 스냅샷보다 현재 디스크와 Git 증거를 우선한다.
3. 기존 변경은 모두 사용자 소유로 보존한다. 현재 작업과 섞지 말고, 필요한 파일만 명시적으로 다룬다.
4. `docs/ROADMAP.md`에서 선행 조건이 충족된 최고 우선순위 항목 하나만 선택한다.

## 제품 목표와 불변 조건

- 번역 가능한 RimWorld 문자열을 안전하게 추출하고 AI 또는 수동 번역, 검수, 원문 변경 이력, 로컬 적용과 RMK 작업 클론 내보내기를 제공한다.
- 사용자 번역, 메모, 상태, 과거 원문과 RMK XLSX 부가 정보를 보존한다.
- Workshop 원본과 RMK 구독본은 읽기 전용이다. 로컬 모드 직접 적용과 RMK 작업 클론 쓰기는 사용자가 UI에서 명시적으로 선택한 대상에만 허용한다.
- 같은 항목의 승계에는 안정적인 파일+키 또는 검증된 `Def Class + Node`를 사용한다. 순번형 ID로 다른 문자열의 번역을 상속하지 않는다.
- XML 키, `defName`, 경로·식별자, 줄바꿈·이스케이프, `{}`·`[]`·`$` 변수, 포맷 토큰, 태그, 문법 접두어와 한국어 자동 조사 문법을 보존한다.
- 원문 변경이 확인되면 기존 번역과 과거 원문은 보존하되 적용 대상에서 제외하고 다시 검토가 필요한 상태로 내린다.
- 번역 허용 Def 필드는 실제 표시 근거가 있어야 한다. 클래스에 따라 의미가 달라지는 `name`, `message`, `symbol` 등을 전역 허용하지 않는다.
- `Patches`는 게임의 최종 병합 결과를 확정할 수 없으므로 일반 Def처럼 자동 번역하지 않는다.
- API 키와 사용자 원문 전체를 설정, 로그, fixture, diff 또는 커밋에 남기지 않는다. 실제 외부 API는 자동 테스트에 사용하지 않는다.

## 저장과 적용 안전성

- 상태와 결과 파일은 임시 파일에 완전히 기록하고 flush·형식 검증 후 같은 파일시스템에서 교체한다.
- 기존 파일을 갱신하는 기능은 복구 가능한 백업 또는 동등한 롤백 지점을 제공해야 한다.
- 손상된 상태를 백업에서 읽었거나 읽지 못했으면 사용자에게 명확히 알린다. 조용히 빈 프로젝트나 정상 상태로 대체하지 않는다.
- 쓰기 기능은 경로가 허용된 루트 안인지 재검증하고, `DryRun` 또는 미리보기, 대상 수, 확인, 실패 요약을 제공한다.
- 실제 사용자 파일을 파괴적 테스트 입력으로 사용하지 않는다.

## 구현 원칙

- 기존 구조와 동작을 먼저 추적하고 가장 작은 복구 가능한 변경을 한다. 전면 재작성이나 UI 프레임워크 교체를 기본 해법으로 삼지 않는다.
- 기능은 성공 경로뿐 아니라 취소, 오류, 부분 성공, 재시도, 저장, 종료와 재시작을 함께 완성한다.
- 성능 최적화는 같은 fixture와 결과 조건의 전후 측정으로 증명하고 정확성·검증·복구 기능을 제거하지 않는다.
- `Start-RimWorldAiReviewGui.ps1`는 UI·상태·오케스트레이션이 결합된 고위험 표면이다. 변경 시 저장, 필터, 선택, 진행, 취소, DPI·테마와 키보드 흐름을 함께 검증한다.
- `Invoke-RimWorldAiTranslation.ps1` 변경 시 추출 범위, ID 매핑, 다중 키 제한, JSON 실패 분할 재시도, 토큰 보존과 부분 성공을 검증한다.
- `Apply-RimWorldAiReviewResults.ps1`와 `Export-RimWorldAiReviewToRmk.ps1` 변경은 실제 적용 전에 fixture `DryRun`, 경로 이탈, 원문 변경, 안전하지 않은 번역과 중복 키를 검증한다.
- `native/`, `launcher/`, `testdata/`, `docs/`에서는 각 디렉터리의 범위 제한 `AGENTS.md`도 따른다.

## 작업 주기와 승인

1. 문제를 코드·로그·재현·측정으로 확인하고 `docs/WORKLOG.md`에 증거와 범위를 기록한다.
2. 완료 조건과 필요한 품질 게이트를 정한 뒤 한 문제만 구현한다.
3. `docs/QUALITY_GATES.md`의 변경 위험에 맞는 검증을 실행한다.
4. 전체 diff를 데이터 무결성, 보안, 실패·재시작, 호환성, UI와 테스트 관점에서 다시 읽는다.
5. 필수 게이트가 통과한 변경만 완료로 판정한다. 실행하지 못한 고위험 검증이 있으면 `BLOCKED` 또는 미검증으로 남긴다.
6. 상태, 결정, 다음 작업을 관련 문서에 갱신한다.

## Git과 외부 작업

- 광범위한 복구·정리 명령과 `git add .`, `git add -A`, 강제 push, amend, 이력 재작성은 금지한다.
- 스테이징과 커밋은 사용자가 요청했거나 현재 자율 작업 범위에 명확히 포함될 때만, 검토한 경로를 명시해 수행한다.
- push, PR, release, 배포, 외부 메시지, 유료 호출, 시스템 변경은 현재 사용자의 명시적 요청 없이는 수행하지 않는다.
- 호스트 불안정이나 저장장치 오류가 재발하면 무거운 빌드와 분석을 중단하고 `docs/WORKLOG.md`에 중단 지점을 남긴다.

## C# 공개 품질 작업 규칙

이 절은 `docs/public-release/` 작업에 적용된다.

### 필수 읽기와 상태 관리

- C# 마이그레이션·공개 품질·검수 작업을 시작하기 전에 `docs/public-release/00_START_HERE.md`, `01_GOVERNANCE_AND_DONE.md`, `02_EXECUTION_PROTOCOL.md`를 읽는다.
- 각 Phase를 시작하기 직전에 해당 Phase 문서만 다시 읽는다.
- 실행 상태의 기준은 `docs/release-readiness/STATE.md`다. 없으면 템플릿으로 생성한다.
- 완료 보고, 과거 테스트 주장, 기존 C# 구현을 사실로 승계하지 말고 현재 코드와 실행 증거로 다시 확인한다.
- 사용자가 설명하지 않아도 기존 PowerShell 안정판과 코드를 직접 조사해 기능·UI 명세를 만든다.

### 기준판과 범위

- Golden Master는 커밋 `4c7d11b49126ba3987e9d49bd16944d4376ba0bc`다.
- 참고용 C# 커밋은 `891d135bb9b37e7d56dd4c29336bb20d277841bc`지만, 현재 로컬 C# 작업이 더 최신이면 현재 작업을 보존하고 감사한다.
- 목표는 기존 기능·UI·작업 파일·번역 요청 동등성을 유지한 C# Windows 앱과 PowerShell 0%의 후보 브랜치다.
- UI 재설계, 기능 축소, 프롬프트 변경, GitHub 언어 비율 조작을 마이그레이션으로 인정하지 않는다.

### 외부 작업과 릴리스 절대 금지

사용자가 현재 대화에서 정확한 대상과 작업을 명시적으로 승인하기 전에는 다음을 수행하지 않는다.

- 원격 push
- PR 생성·수정·병합
- 태그 생성·이동·삭제
- GitHub Release 생성·수정·삭제
- 기존 Release asset 삭제·교체·재업로드
- 저장소 설명·Topics·원격 README 변경
- GitHub Actions 수동 실행
- 외부 파일 업로드
- 실제 공개 배포
- 유료 API 호출
- 컴퓨터 종료·재시작·절전 예약

기존 `v1.0.0`은 immutable로 취급한다. 로컬 `dotnet publish`와 로컬 ZIP은 검증 목적으로만 허용한다.

### Git과 사용자 데이터 보호

- `git reset --hard`, `git clean -fd[x]`, 광범위한 checkout/restore, 강제 push, amend, 공개 이력 재작성, `git add .`, `git add -A`를 사용하지 않는다.
- 사용자 승인 없는 커밋을 만들지 않는다.
- 기존 staged·unstaged·untracked 파일은 사용자 소유로 보존한다.
- 실제 `%LOCALAPPDATA%`, Workshop, RMK 구독본과 사용자 작업 원본을 테스트에 쓰거나 수정하지 않는다.
- 테스트는 익명화 복사본 또는 합성 fixture와 고유한 임시 디렉터리만 사용한다.
- 실제 API 키, 인증 헤더, 사용자 원문 전체를 로그·fixture·diff·보고서에 남기지 않는다.

### 구현과 검증 원칙

- 먼저 계획과 기능 대응표를 만들고, 기능 ID 단위로 구현·테스트·증거를 닫는다.
- 첫 빌드 오류나 테스트 실패에서 멈추지 말고 원인을 수정한 뒤 관련 회귀를 다시 실행한다.
- PowerShell 코드를 하나의 거대한 C# Form·Window·클래스로 직역하지 않는다.
- UI, 상태, 업무 로직, 저장, 파일 시스템, HTTP를 분리하고 핵심 로직은 UI 없이 테스트 가능하게 한다.
- 긴 작업은 비동기·취소 가능해야 하며 UI 스레드를 차단하지 않는다.
- 기존 작업 파일은 읽기 전용 열기, round-trip, 실패 주입, 백업·롤백으로 검증한다.
- 실제 API 없이 canonical request payload와 fake response 결과를 기준판과 비교한다.
- 보안·성능·패키지 검사는 최종 C# 소스와 로컬 publish 결과물에 다시 수행한다.
- 테스트를 삭제·완화하거나 기능을 제거해 PASS를 만들지 않는다.

### 완료 판정

최종 판정은 `PASS`, `FAIL`, `BLOCKED` 중 하나다.

- 증거 없는 항목은 PASS가 아니다.
- 빌드 성공만으로 완료가 아니다.
- 필수 기능 누락, 데이터 손실 위험, API 키 노출, Critical/High 취약점, PowerShell 실행 의존성, 중대한 UI 차이, 실패·취소 후 상태 손상, 중대한 성능 회귀가 있으면 PASS 금지다.
- 사용자 수동 UI 확인과 깨끗한 PC 검사가 남으면 그 항목은 명확히 `BLOCKED` 또는 사용자 승인 대기로 기록한다.
- 공개 GitHub 상태를 변경하지 않았음을 최종 보고에 반드시 포함한다.
